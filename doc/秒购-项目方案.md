# 秒购（MiaoGo）— 高并发电商秒杀交易平台

> 一句话定位：一套从零搭建、支撑限时秒杀与限量发售等高并发营销活动的电商交易系统，核心目标是**在流量尖峰下保证不超卖、不丢单、数据库不被打垮**。

---

## 一、项目背景与简历素材

### 1.1 业务背景（编一个站得住脚的故事）

某新零售/电商团队主营快消与潮流商品，依赖「整点秒杀」「限时抢购」「限量发售」等营销玩法做拉新和促活。原有单体系统在大促整点会集中爆发流量，频繁出现以下问题：

- 库存超卖、扣减错乱，售后赔付严重；
- 整点瞬时流量把数据库连接打满，正常用户也无法下单；
- 订单创建与支付链路耦合，下游一个服务抖动就拖垮整个下单；
- 订单超时未支付不能及时释放库存，库存利用率低；
- 历史订单数据量过亿，订单查询越来越慢。

本项目从零搭建一套独立的高并发交易系统来承接这些营销活动，逐步从单体演进到微服务架构，重点解决**高并发下的数据一致性、性能与可用性**问题。

> 说明：这是个人练手项目，上面的"线上故障"是为了给每项技术一个真实的引入理由。简历上可如实写为"个人项目 / 高并发场景模拟与压测验证"，不必伪装成线上生产系统——压测数据本身就是最好的支撑。

### 1.2 项目名称

- 中文名：**秒购**
- 英文代号：**MiaoGo**（或 `seckill-trade-platform`）
- Slogan：*整点不超卖，尖峰不宕机。*

> 名字可替换。如果想更有"潮品发售"的故事感，可改成 **潮汐 / DropX**（主打限量球鞋、演出票务等强秒杀场景），背景叙事同理。

### 1.3 项目概述（简历段落版，可直接用）

> **秒购（MiaoGo）· 高并发电商秒杀交易平台 ｜ 个人项目**
>
> 独立设计并实现一套面向限时秒杀/限量发售场景的电商交易系统，采用 Spring Cloud Alibaba 微服务架构。通过"单体起步—压测暴露瓶颈—针对性引入中间件"的演进式开发，系统性解决了高并发下的超卖、缓存一致性、流量削峰、数据扩容等问题。经 JMeter 压测，秒杀链路可在 **单机 5k+ QPS** 下保证零超卖，核心接口 P99 延迟控制在 **100ms 内**，数据库写入峰值经 MQ 削峰后下降约 **80%**。

### 1.4 简历要点版（bullet，挑你真正做到的写）

- 基于 Redis + Lua 原子预扣库存 + Redisson 分布式锁，解决秒杀超卖问题，压测 1w 并发抢购零超卖；
- 设计多级缓存（Caffeine 本地缓存 + Redis 集群），结合空值缓存、布隆过滤器、逻辑过期，解决缓存穿透/击穿/雪崩，热点商品缓存命中率 99%+；
- 引入 RocketMQ 实现下单异步化与服务解耦，秒杀写请求削峰后数据库写入峰值下降约 80%；基于事务消息保证库存与订单最终一致；
- 使用 RocketMQ 延迟消息实现订单超时自动取消与库存回补，替代轮询定时任务，库存释放延迟 < 1s；
- 基于 ShardingSphere 对订单库按 user_id 分库分表，并通过 binlog/MQ 同步商家维度冗余表，解决跨维度查询问题；
- 通过 Nacos / Gateway / Sentinel / Seata 完成微服务治理，实现限流、熔断降级与分布式事务；
- 订单状态流转采用状态机模式，下单校验采用责任链模式，提升可维护性与扩展性。

### 1.5 电梯陈述（跟人口头交流用，30 秒）

"这是一个高并发电商秒杀系统。我没有上来就堆架构，而是先写了个能跑的单体，然后用 JMeter 把它压垮——亲眼看到超卖、慢查询、数据库被打满。每出现一类问题，我才引入对应的技术去解决：超卖用 Redis 预扣加分布式锁，读压力用多级缓存，写尖峰用消息队列削峰，数据量大了用分库分表，最后拆成微服务做治理。整个过程的重点不是'我用了哪些中间件'，而是'每个中间件解决了我亲手遇到的什么问题、有哪些方案、为什么这么选'。"

### 2.2 技术 → 真实场景映射

| 技术 | 在哪个场景下"非它不可" |
|---|---|
| 事务 / MVCC | 下单原子性（扣库存+建订单+扣余额）；并发扣库存时观察丢失更新、不可重复读，理解 RR 下 MVCC 与间隙锁 |
| 乐观锁/悲观锁/分布式锁 | 秒杀超卖逐层治理；优惠券"每人限领一张"；下单幂等 |
| 数据库索引 | 订单列表慢查询 → explain、联合索引、覆盖索引、最左前缀、深分页优化 |
| 分库分表 | 订单过亿 → 按 user_id 分片，再用冗余表解决商家维度查询 |
| 缓存 / Redis 集群 | 多级缓存、库存预扣（Lua 原子）、穿透/击穿/雪崩、热 key |
| 消息队列 | 削峰、解耦、事务消息最终一致、延迟队列超时取消 |
| 线程池 | 异步落库埋点、CompletableFuture 并行拼装详情页、自定义线程池 |
| 定时任务 | 订单超时取消回补库存；单机 → 集群重复执行的坑 → XXL-JOB / 延迟队列 |
| 微服务管理 | 服务拆分、Nacos、Gateway、Sentinel 限流熔断降级、Seata、链路追踪 |
| 设计模式 | 订单状态机（状态）、支付/促销（策略）、下单校验风控（责任链）、订单事件（观察者）、分片策略（策略） |

## 三、核心理念

1. **问题驱动，而非技术驱动。** 永远先制造出"非用不可的痛点"，再引入技术。不要为了用 Redis 而用 Redis。
2. **单体优先，渐进演进。** 先把单体做到能跑，用压测压垮它，在它垮掉的地方引入中间件。不要一上来就微服务+分库分表。
3. **每一步都要压测验证。** 只有压测能让你"看见"超卖、慢查询、削峰、限流是否真的生效。没有压测的高并发代码都是纸上谈兵。
4. **每解决一个问题就写一篇笔记。** 记录"现象 → 排查 → 几种方案 → 为什么这么选"。面试答案是从踩过的坑里长出来的。

## 四、分阶段开发路线（含 Checklist 与验收标准）

> 每个 Phase 的结构：**目标 → 要主动制造的问题 → 任务清单 → 验收标准**。
> 验收标准尽量是**可压测、可观测、可量化**的，做完打勾。

### Phase 0 — 单体跑通业务（打地基）

**目标：** 不谈性能，先让业务"对"。搭好单体骨架，跑通正常下单全流程。

**要主动制造的问题：** 暂无，先建立基线。

**任务清单：**

- [ ] 搭建 Spring Boot 单体工程，规划包结构（controller / service / mapper / domain）
- [ ] 设计核心表：用户、商品、库存、订单、订单明细、支付流水（画 ER 图）
- [ ] 实现用户、商品、库存的基础 CRUD
- [ ] 实现正常下单流程：校验 → 扣库存 → 建订单 → 扣余额，包在一个本地事务里
- [ ] 用**状态模式**实现订单状态机（待支付/已支付/已取消/已完成），明确状态流转规则
- [ ] 用**责任链模式**实现下单前置校验（库存校验、限购校验、用户状态校验）
- [ ] 配置一个自定义 `ThreadPoolExecutor`，用于异步写操作日志/埋点
- [ ] 为下单核心链路写接口测试

**验收标准：**

- [ ] 正常下单链路全程跑通，数据落库正确
- [ ] **事务回滚验证**：人为让"扣余额"抛异常，确认库存和订单一并回滚，无脏数据
- [ ] 非法状态流转被拒绝（如"已取消"订单不能再支付）
- [ ] 能说清状态机/责任链相比 if-else 堆砌的好处

---

### Phase 1 — 制造并发，治理超卖（核心中的核心）

**目标：** 亲手制造超卖，再逐层解决，吃透各种锁与 MVCC。

**要主动制造的问题：** 用 JMeter 高并发抢购少量库存，观察超卖。

**任务清单：**

- [ ] 写秒杀下单接口（先用 Phase 0 的朴素扣库存逻辑）
- [ ] 用 JMeter 配置 1000 并发抢 100 库存，**复现超卖**，记录现象
- [ ] 方案一：数据库**乐观锁**（version 字段 / `where stock > 0`），压测验证
- [ ] 方案二：数据库**悲观锁**（`select ... for update`），压测验证
- [ ] 方案三：Redis 预扣库存 + **Lua 脚本**保证原子性，压测验证
- [ ] 方案四：**Redisson 分布式锁**（理解看门狗续期），压测验证
- [ ] 实现下单**幂等**（防止用户重复提交：唯一索引 / 分布式锁 / token）
- [ ] 实现优惠券"每人限领一张"（分布式锁 vs 数据库唯一键对比）
- [ ] 用 MySQL 实验观察 **MVCC**：RR 隔离级别下的快照读 vs 当前读、不可重复读现象

**验收标准：**

- [ ] 1000 并发抢 100 库存，**最终恰好售出 100 件，零超卖、零少卖**
- [ ] 产出一张**方案对比表**：每种方案的 QPS、正确性、优缺点、适用场景
- [ ] 能解释清楚乐观锁/悲观锁/分布式锁/Redis 预扣各自的取舍
- [ ] 能讲清 MVCC 是怎么实现"读不加锁"的，RR 与 RC 的区别
- [ ] 重复提交同一订单只生成一条记录

---

### Phase 2 — 加缓存（扛读压力）

**目标：** 用缓存扛住读流量，吃透缓存三大问题与一致性。

**要主动制造的问题：** 压测商品详情接口，看数据库读 QPS 被打高；删掉缓存看击穿；查不存在的 ID 看穿透。

**任务清单：**

- [ ] 商品详情接入 Redis 缓存（Cache Aside 模式）
- [ ] 加 **Caffeine 本地缓存**做 L1，构成多级缓存
- [ ] 防**穿透**：缓存空值 / 布隆过滤器
- [ ] 防**击穿**：热点 key 互斥重建 / 逻辑过期
- [ ] 防**雪崩**：过期时间加随机抖动 / 缓存预热
- [ ] 实现缓存一致性，至少对比两种方案：延迟双删 vs binlog（Canal）订阅同步
- [ ] 秒杀开始前**缓存预热**（活动商品与库存提前加载到 Redis）

**验收标准：**

- [ ] 商品详情接口压测，**缓存命中率 > 95%**，数据库读 QPS 显著下降
- [ ] 模拟穿透（狂查不存在 ID），数据库不被打穿
- [ ] 模拟击穿（手动删热点 key 后高并发），数据库读请求收敛到 1～几次重建
- [ ] 能讲清三种缓存问题的区别与各自解法，以及缓存一致性方案的取舍

---

### Phase 3 — 引入消息队列（扛写 + 解耦）

**目标：** 用 MQ 削峰、解耦、做延迟队列，吃透 MQ 的可靠性问题。

**要主动制造的问题：** 秒杀瞬时写请求把数据库写入打满；下游通知服务抖动拖垮下单。

**任务清单：**

- [ ] 引入 RocketMQ
- [ ] 秒杀下单**异步化**：Redis 预扣成功后发消息，消费者异步落库（削峰）
- [ ] 下单成功事件**解耦**下游：积分、优惠券核销、通知（发布订阅）
- [ ] 用**事务消息**保证"扣库存"与"建订单"最终一致
- [ ] 消费端做**幂等**（防重复消费），处理消息丢失、消息堆积
- [ ] 配置**死信队列**处理消费失败
- [ ] 用 **RocketMQ 延迟消息**实现订单超时自动取消 + 库存回补

**验收标准：**

- [ ] 秒杀写请求经 MQ 削峰后，**数据库写入峰值明显下降**（记录削峰前后曲线）
- [ ] 下游通知服务故意挂掉，主下单链路**不受影响**仍能成功
- [ ] 重复投递同一条消息，业务只生效一次（幂等验证）
- [ ] 订单超时后**库存在 1s 内自动回补**
- [ ] 能讲清消息丢失/重复/堆积/顺序的成因与对策，事务消息原理

---

### Phase 4 — 数据扩容（分库分表 + 搜索）

**目标：** 解决数据量过大，亲身踩分片键与跨维度查询的坑。

**要主动制造的问题：** 灌入千万级订单数据，订单列表查询变慢；按商家维度查询时分片全扫。

**任务清单：**

- [ ] 用脚本灌入千万级订单测试数据
- [ ] 优化慢查询：`explain` 分析、补**联合索引/覆盖索引**、解决**深分页**
- [ ] 引入 ShardingSphere，订单库按 **user_id 分库分表**
- [ ] 暴露问题：按 **merchant_id 查询**时跨所有分片，性能差
- [ ] 解决：建商家维度**冗余表**（按 merchant_id 分片），用 binlog/MQ 同步
- [ ] 处理分布式环境下的**全局唯一 ID**（雪花算法 / 号段模式）
- [ ] 商品搜索接入 **Elasticsearch**，理解倒排索引与数据同步

**验收标准：**

- [ ] 慢查询优化后，订单列表查询耗时大幅下降（记录优化前后 ms 数）
- [ ] 能解释**为什么选 user_id 作分片键**，以及它带来的跨维度查询代价
- [ ] 用户维度与商家维度查询都能高效命中各自的分片
- [ ] 能讲清深分页为什么慢、覆盖索引为什么快、最左前缀原则
- [ ] 商品搜索走 ES，能说清与 MySQL `like` 的区别

---

### Phase 5 — 拆微服务（治理）

**目标：** 把单体拆成微服务，引入完整治理体系，理解"拆"带来的新麻烦。

**要主动制造的问题：** 单机 `@Scheduled` 在多实例下重复执行；跨服务调用后事务不再是本地事务；某服务雪崩拖垮全局。

**任务清单：**

- [ ] 按域拆分：网关 / 用户 / 商品 / 库存 / 订单 / 支付 / 营销 / 搜索
- [ ] 接入 **Nacos** 做注册中心与配置中心
- [ ] 接入 **Spring Cloud Gateway** 统一入口（鉴权、路由）
- [ ] **OpenFeign** 做服务间调用
- [ ] **Sentinel** 做限流、熔断、降级（秒杀接口限流尤其关键）
- [ ] 跨服务下单用 **Seata** 保证分布式事务（对比 AT / TCC / Saga / 消息最终一致）
- [ ] 暴露定时任务重复执行问题，换成 **XXL-JOB** 分布式调度
- [ ] 接入 **SkyWalking** 做链路追踪

**验收标准：**

- [ ] 服务可独立部署、多实例运行，Nacos 上正常注册与发现
- [ ] 秒杀接口被 Sentinel 限流，超出阈值的请求被快速拒绝/排队而非压垮后端
- [ ] 故意让某服务超时，熔断降级生效，不发生雪崩
- [ ] 跨服务下单异常时，分布式事务正确回滚，数据一致
- [ ] 多实例下定时任务**只执行一次**
- [ ] 能讲清各分布式事务方案的取舍

### Phase 6 — 可观测 + 全链路压测闭环（收口）

**目标：** 把监控、日志、压测补齐，让前面所有优化"看得见"。

**任务清单：**

- [ ] 接入 **Prometheus + Grafana**，监控 QPS、RT、错误率、线程池、连接池、缓存命中率
- [ ] 接入 **ELK** 统一日志，支持按 traceId 检索全链路日志
- [ ] 用 JMeter 做**全链路压测**，画出各 Phase 优化前后的对比报告
- [ ] 自定义线程池/连接池**监控埋点**，观察是否打满、拒绝
- [ ] 整理一份**性能优化报告**（每个瓶颈点、解法、量化收益）

**验收标准：**

- [ ] Grafana 大盘能实时看到核心指标
- [ ] 出问题时能凭 traceId 在 Kibana 串起全链路日志
- [ ] 产出一份带数据的性能优化报告（削峰曲线、QPS 提升、RT 下降、缓存命中率等）
- [ ] 能用一句话量化每项技术带来的收益（这就是简历里的数字）

## 五、贯穿全程的工程实践

- **Git 协作**：GitHub Flow 分支模型 + Conventional Commits 提交规范，每个 Phase 拉 feature 分支。
- **技术笔记**：每解决一个问题写一篇，模板为「现象 → 排查过程 → 候选方案 → 选型理由 → 复盘」。这是你最值钱的产出。
- **压测先行**：每个 Phase 引入技术前后都压测对比，没有数据的优化等于没做。
- **配置安全**：密钥（如第三方 API key）放配置中心或本地配置文件，不进 Git。

## 六、避坑与 Scope 提醒

- **别贪快跳阶段。** 每个 Phase 的价值在于"先体验问题"。直接抄成品架构，你还是没遇到问题，等于白做。
- **scope 要现实。** 这是几个月的项目，不是几周。一个人的玩具项目"本不需要"分库分表和微服务——你是在用压测**人为制造负载**来模拟大厂场景、理解解决方案，心里要清楚这层，面试也可以坦诚这么说，反而显得清醒。
- **数字要诚实。** 简历里的 QPS、命中率、削峰比例都来自你自己的压测报告，能复现、能解释。别写自己讲不清的数字。
- **深度优先于广度。** 把秒杀超卖这一个点吃到极致（六七种方案随口讲），比浅尝十个中间件更能打动面试官。

## 七、差异化建议（简历加分项）

这个秒杀电商骨架虽然经典，但也烂大街。想让它不像教程克隆：

- **换个有故事的场景**：把"秒杀"换成限量球鞋发售、演唱会抢票、医院挂号——强秒杀属性更强，叙事更生动。
- **加一个 AI 模块**：智能导购 / 会话式搜索 / 商品摘要标签（你正好在用 AI 编码工具，可顺手接大模型 API），让它有差异化亮点。
- **沉淀一篇博客或一份压测报告**：把"如何把超卖从 N 种方案打到零"写成文章，比简历一行字更能体现深度。
